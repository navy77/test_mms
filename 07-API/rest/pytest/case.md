# 🧪 Pytest Cases Specification

เอกสารอธิบายชุดกรณีการทดสอบ (Test Cases) สำหรับ REST API Service ของระบบประวัติประมวลผลข้อมูล ซึ่งครอบคลุมการทำงาน 3 ส่วนหลัก ได้แก่ **Status**, **Alarm** และ **Data (History)**

---

## 1. Status Test Cases (`pytest/test_status.py`)
ชุดการทดสอบโมดูลการคำนวณสัดส่วนและระยะเวลาสถานะของอุปกรณ์ (แบบต่อเนื่องกัน):
* **Case 1 (Simple Sequence):** ทดสอบการเปลี่ยนสถานะปกติแบบต่อเนื่อง (เช่น `status_1` ➡️ `status_2`)
* **Case 2 (Initial Status):** ทดสอบกรณีอุปกรณ์มีสถานะเริ่มต้นสืบทอดสะสมมาจากเวลาก่อนหน้าเริ่มสืบค้น แล้วมาเปลี่ยนสถานะภายในช่วงเวลาสืบค้น
* **Case 3 (No events, no initial):** ทดสอบกรณีที่ไม่มีข้อมูลประวัติสถานะใดๆ เลย ทั้งก่อนหน้าและระหว่างการสืบค้น (ต้องแสดงเป็นสถานะ `"no data"` ตลอดทั้งช่วงสืบค้น)
* **Case 4 (No events, initial present):** ทดสอบกรณีไม่มีเหตุการณ์เปลี่ยนสถานะใหม่เลยตลอดช่วงสืบค้น แต่มีสถานะเริ่มต้นสะสมมาก่อนหน้า (สถานะนั้นต้องยาวครอบคลุมเป็น 100% ของช่วงสืบค้น)

---

## 2. Alarm Test Cases (`pytest/test_alarm.py`)
ชุดการทดสอบโมดูลการคำนวณสัญญาณเตือน ซึ่งมาคู่กัน (เริ่ม-จบ) และตัดผลลัพธ์ของ `"normal"` และ `"no data"` ออกจากการคำนวณ:
* **Case 1 (Simple Pair):** ทดสอบคู่สัญญาณเตือนมาตรฐาน (เริ่ม `alarm_1` ➡️ สิ้นสุดด้วย `alarm_1_`)
* **Case 2 (Carry-in / Alarm starts before start time):** สัญญาณเตือนเริ่มสะสมมาก่อนหน้าเวลาสืบค้น และมาสิ้นสุด (`alarm_1_`) ภายในช่วงสืบค้น
* **Case 3 (Overlapping Alarms):** สัญญาณเตือนเกิดขึ้นซ้อนทับกันหลายตัวในเวลาเดียวกัน (คำนวณและเก็บระยะเวลาสะสมแยกอิสระของแต่ละสถานะ)
* **Case 4 (No data):** ไม่มีข้อมูลหรือสถานะแจ้งเตือนใดๆ เกิดขึ้นเลย (ผลลัพธ์ที่ได้คืนต้องเป็นอาเรย์ว่าง `[]`)
* **Case 5 (Close event without preceding start event):** ตรวจพบสัญญาณสิ้นสุด (เช่น `alarm1_`) ในช่วงสืบค้น แต่ไม่มีสถานะเริ่มต้นสะสมก่อนหน้า (ระบบย้อนกลับไปคิดประวัติสะสมตั้งแต่เวลาเริ่มสืบค้น `07:00:00` ไปจนถึงเวลาที่มีสัญญาณสิ้นสุด)
* **Case 6 (Start event without preceding end event before query end):** สัญญาณเตือนเริ่มทำงานในช่วงสืบค้น และไม่มีสัญญาณสิ้นสุดเข้ามาปิดจนสิ้นช่วงการดึงประวัติ (นับเวลาจากเวลาเกิดสัญญาณเตือนไปจนสุดขอบเขตเวลาปัจจุบัน/สืบค้น)

## 3. Device Test Cases (`pytest/test_device.py`)
ชุดการทดสอบโมดูลการคำนวณสัดส่วนและระยะเวลาสถานะการออนไลน์/ออฟไลน์ของเครื่องจักร (จากตาราง `device_tb`):
* **Case 1 (Simple Sequence):** ทดสอบการเปลี่ยนสถานะการทำงานปกติ (เช่น `online` ➡️ `offline`)
* **Case 2 (Initial Status):** ทดสอบกรณีอุปกรณ์มีสถานะเริ่มต้นสะสมมาก่อนหน้าเริ่มสืบค้น (เช่น `online`) แล้วเปลี่ยนเป็น `offline` ภายในช่วงเวลาสืบค้น
* **Case 3 (No events, no initial):** ทดสอบกรณีไม่มีข้อมูลประวัติสถานะของอุปกรณ์เลย (ต้องแสดงเป็นสถานะ `"no data"` ตลอดช่วงสืบค้น)
* **Case 4 (No events, initial present):** ทดสอบกรณีไม่มีข้อมูลเหตุการณ์เปลี่ยนสถานะใหม่เลย แต่มีสถานะเริ่มต้นสะสมค้างอยู่ก่อน (สถานะนั้นต้องยาวครอบคลุม 100% ของช่วงสืบค้น)

---

## 4. Data Test Cases (`pytest/test_data.py`)
ชุดการทดสอบโมดูลดึงข้อมูลดิบรายชั่วโมง/รายวัน/รายเดือน และการจัดการขอบเขตเวลาทำงาน (Production Day):
* **Case 1 (Production Day Boundary - After 07:00):** เคสเวลาหลัง 07:00 น. ➡️ ตรวจสอบว่าช่วงการคำนวณวันผลิตต้องเป็นวันปัจจุบัน
* **Case 2 (Production Day Boundary - Before 07:00):** เคสเวลาช่วงเช้ามืดก่อน 07:00 น. ➡️ ตรวจสอบว่าช่วงการคำนวณวันผลิตต้องย้อนหลังกลับไป 1 วัน
* **Case 3 (Hourly Process Query Success):** ทดสอบยิง API เรียกข้อมูลประวัติรายชั่วโมงของ process โดยการ Mock ClickHouse Client และคืนค่าข้อมูลถูกต้องสำเร็จ (200 OK)
* **Case 4 (Hourly Process Query Not Found):** จำลองสถานการณ์ไม่มีข้อมูลใน ClickHouse สำหรับชื่อ process นั้นๆ และคืนสถานะข้อผิดพลาด (400 Bad Request / Item not found)
* **Case 5 (Monthly Invalid Parameters):** ทดสอบความถูกต้องในการรับค่า API โดยระบุพารามิเตอร์ผิดพลาด (เช่น เดือน 13) และคืนข้อผิดพลาด (422 Unprocessable Entity จาก FastAPI)
